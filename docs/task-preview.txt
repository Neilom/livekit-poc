voice-calling-livekit.

[2/11] [VCL] Rota POST para iniciar ligação (chatId → sala + bridge + token)
--------------------------------------------------------------------------------
Story Points: 5
Prioridade: Alto
Depende de: —
Paraleliza com: 5 (VCA - Lambda S3)

Descrição:
Criar a rota principal do VCL que recebe o chatId, cria a sala LiveKit,
monta o bridge de áudio (werift ↔ rtc-node), envia o SDP offer para o VCA
pela rota existente e retorna o token para o browser.

Escopo técnico:
- Nova rota POST (ex: POST /call) — body: { chatId }
- Validar presença do chatId
- Criar room no LiveKit com o chatId como nome (ou prefixo: call-{chatId})
  usando createRoom() de livekit-manager.ts
- Chamar createBridge(roomName) que:
  - Conecta bridge-agent ao room (rtc-node)
  - Cria PeerConnection werift + SDP offer
  - Aguarda ICE gathering (~10s timeout)
- Enviar sdpOffer para o VCA pela rota já existente (POST existente no VCA
  para iniciar chamada Meta)
- Gerar token JWT para o browser (generateToken com identity "browser-agent")
- Salvar sessão (chatId → CallSession com bridge handle, roomName, status)
- Retornar: { token, livekitUrl } (ou o necessário para o web-app conectar)

Referência do código existente (POC):
- server.ts: POST /call/start (linhas 32-78) — adaptar para receber chatId
  em vez de "to"
- call-bridge.ts: createBridge() — reutilizar quase inteiro
- livekit-manager.ts: createRoom(), generateToken() — sem mudanças

O que muda em relação ao POC:
- Entrada é chatId (não "to" com número de telefone)
- VCL não chama Meta diretamente; envia offer para VCA que já tem a rota
- Não retorna sdpOffer nem metaCallId para o front (fica interno)

Critérios de aceite:
- POST com chatId cria sala e retorna token válido
- Bridge-agent aparece no room do LiveKit
- SDP offer é enviado ao VCA com sucesso
- Sessão fica como "waiting_answer" até receber o SDP answer via SNS

[3/11] [VCL] Listener SNS — receber SDP answer e iniciar gravação
--------------------------------------------------------------------------------
Story Points: 5
Prioridade: Alto
Depende de: 2
Paraleliza com: 5 (VCA - Lambda S3)

Descrição:
O VCA recebe o webhook da Meta com o SDP answer e publica no SNS de statuses.
O VCL escuta esse tópico SNS e, quando chega o answer, aplica no bridge para
completar a conexão WebRTC e inicia a gravação via Egress.

Escopo técnico:
- Configurar subscriber SNS no VCL (HTTP endpoint ou SDK pull — alinhar
  padrão do time)
- Ao receber status com SDP answer:
  - Identificar a sessão pelo chatId / metaCallId do evento
  - Chamar bridge.applyAnswer(sdpAnswer) — normaliza SDP e seta
    remoteDescription no werift PeerConnection
  - Marcar sessão como "active"
- Iniciar gravação: chamar startRecording(roomName, callId)
  - Egress: RoomCompositeEgress → arquivo .ogg no S3
  - Usar IAM Role do projeto (não credenciais S3 hardcoded)
  - Ajustar egress-manager.ts para NÃO passar accessKey/secretKey no
    S3Upload — apenas region e bucket; o Egress container assume a Role

Referência do código existente (POC):
- server.ts: POST /call/:callId/answer (linhas 81-112) — lógica de
  applyAnswer + startRecording; agora disparado pelo SNS em vez de POST
- egress-manager.ts: startRecording() — remover S3Upload com credenciais
  explícitas, usar apenas { region, bucket } para Role-based auth

Critérios de aceite:
- SDP answer chega via SNS e é aplicado automaticamente (sem passo manual)
- Conexão WebRTC entre bridge e Meta se estabelece (áudio bidirecional)
- Gravação inicia em S3 no path recordings/{callId}/...
- Egress usa IAM Role (sem accessKey/secretKey no código)

[5/11] [VCL] Rota POST para terminar ligação (chatId)
--------------------------------------------------------------------------------
Story Points: 3
Prioridade: Alto
Depende de: 2
Paraleliza com: 4, 6

Descrição:
Rota que recebe chatId e limpa tudo no lado LiveKit: para gravação,
termina bridge, deleta sala.

Escopo técnico:
- Nova rota POST /call/terminate — body: { chatId }
- Chamar cleanupSession(chatId, "agent_hangup")
  (ver seção "Cleanup e Race Conditions")
- Se cleanupSession retornar false (sessão não existe ou já terminada):
  retornar 404 { error: "Session not found or already terminated" }
- Se retornar true: retornar 200 { status: "terminated" }

Nota: O VCL NÃO chama o VCA para terminar a Meta — o web-app faz isso
diretamente na rota existente do VCA.

Nota: cleanupSession é a mesma função usada pela tarefa 6 (SNS call_ended),
garantindo que a lógica de cleanup está centralizada e idempotente.

Referência do código existente (POC):
- server.ts: POST /call/:callId/terminate (linhas 115-153) — extrair
  lógica para cleanupSession() e adaptar para buscar por chatId

Critérios de aceite:
- Gravação para e arquivo .ogg é finalizado no S3
- Bridge werift é fechado, room LiveKit deletado
- Sessão removida do store
- Chamada a cleanupSession com sessão já terminada retorna 404 (idempotente)
- GET /call/status?chatId=xxx retorna 404 após terminate

[9/11] [VCA] Lambda S3 — salvar URL do áudio na mensagem
--------------------------------------------------------------------------------
Story Points: 5
Prioridade: Médio
Depende de: 11 (S3 precisa estar configurado com trigger)
Paraleliza com: —

Descrição:
Quando o Egress finaliza a gravação e salva o .ogg no S3, um evento S3
dispara uma Lambda no VCA que salva a URL do áudio na mensagem
correspondente.

Escopo técnico:
- Criar Lambda function no contexto do VCA
- Runtime: Node.js 20 (padrão do VCA)
- Trigger: S3 ObjectCreated no bucket, prefix: recordings/
- A Lambda recebe o evento S3, extrai:
  - bucket: event.Records[0].s3.bucket.name
  - key: event.Records[0].s3.object.key
    ex: recordings/abc-123/call-abc-123-2025-01-15T103000.ogg
  - chatId: key.split("/")[1]  → "abc-123"
- Gerar presigned URL com validade de 7 dias:
  - Usar getSignedUrl do @aws-sdk/s3-request-presigner
  - A Lambda precisa de IAM Role com s3:GetObject no bucket
- Buscar mensagem de ligação no banco do VCA pelo chatId:
  - Query: mensagem do tipo "call" com chatId = "abc-123" e sem audioUrl
  - Se não encontrar: log error e retornar (não falhar a Lambda)
- Atualizar mensagem com campo audioUrl (presigned URL)
- Schema de mensagens — campo novo:
  - audioUrl: string | null (nullable, default null)
  - Requer migration no banco do VCA (ALTER TABLE ou equivalente)

Decisão sobre URL:
- Presigned URL com 7 dias de validade (renovável por um endpoint futuro
  se necessário)
- Não usar URL pública (risco de segurança)
- Não precisar de CORS no S3 (presigned URL funciona sem CORS para
  download direto via <audio src>)

Critérios de aceite:
- Arquivo novo no S3 → Lambda é invocada
- Lambda extrai chatId do path e encontra a mensagem
- audioUrl (presigned, 7 dias) é salva na mensagem em < 30s
- Erros logados (msg não encontrada, formato inesperado, etc.)
- Lambda é idempotente (se rodar 2x para o mesmo arquivo, não duplica)

[10/11] [infra] Tasks ECS separadas: LiveKit Server, Egress e VCL
--------------------------------------------------------------------------------
Story Points: 5
Prioridade: Alto
Depende de: —
Paraleliza com: 11

Descrição:
Cada componente precisa escalar de forma independente. O LiveKit Server,
o Egress worker e o VCL devem rodar em task definitions separadas no ECS.

Escopo técnico:
- Task definition 1: LiveKit Server (livekit/livekit-server)
  - Portas: 7880 (WS), 7881 (TCP), 7882 (UDP/WebRTC)
  - Config via livekit.yaml montado (EFS ou inline no task definition)
  - Depende de Redis (ElastiCache)
  - IMPORTANTE: Precisa de EC2 launch type (não Fargate) para suporte
    a UDP e network mode host. Alternativamente, usar Fargate com NLB
    para UDP — consultar DevOps sobre viabilidade
  - O LiveKit Server precisa de IP público (ou NLB público) para que
    browsers externos conectem via WebRTC
  - Env vars: LIVEKIT_CONFIG (YAML inline) ou volume mount
- Task definition 2: Egress (livekit/egress)
  - Precisa de acesso ao LiveKit (ws_url) e Redis
  - IAM Task Role com s3:PutObject no bucket de recordings
  - cap_add: SYS_ADMIN (Chrome headless interno do Egress)
  - IMPORTANTE: SYS_ADMIN capability requer EC2 launch type (não
    disponível em Fargate). Confirmar com DevOps
  - network_mode: host (como no docker-compose) — requer EC2
- Task definition 3: VCL (voice-calling-livekit)
  - Pode rodar em Fargate (não tem requisitos especiais)
  - Acesso ao LiveKit (WS interno), VCA (HTTP), SNS subscription
  - Health check: GET /health (adicionar rota simples no server.ts)
  - Env vars: LIVEKIT_URL, LIVEKIT_API_KEY, LIVEKIT_API_SECRET,
    VCA_BASE_URL, VCA_API_TOKEN, SNS_TOPIC_ARN, S3_REGION, S3_BUCKET
- Service discovery:
  - LiveKit Server: registrar no Cloud Map (AWS Service Discovery)
    Nome interno: livekit.voice.internal → resolve para IP do container
  - O VCL e Egress usam esse DNS para LIVEKIT_URL
  - Alternativa: ALB/NLB interno com DNS fixo
  - Consultar DevOps para definir padrão
- Redis: Usar ElastiCache existente se disponível. Se não, criar
  cluster Redis dedicado (single-node para MVP, cluster para prod).
  O LiveKit e Egress usam o mesmo Redis.
- Security Groups:
  - LiveKit: inbound 7880 (WS) de VCL + browsers, 7881 (TCP) interno,
    7882 (UDP) de internet (WebRTC)
  - Egress: outbound para LiveKit, S3, Redis
  - VCL: inbound 3000 (HTTP) de ALB, outbound para LiveKit, VCA, SNS

Critérios de aceite:
- Cada componente escala independentemente
- LiveKit acessível pelo VCL e pelo browser (URL pública para WebRTC)
- Egress conecta no LiveKit e grava no S3
- VCL conecta no LiveKit, no VCA e no SNS
- Health checks funcionais para cada task

[11/11] [infra] Configurar S3 para Egress + trigger Lambda do VCA
--------------------------------------------------------------------------------
Story Points: 3
Prioridade: Alto
Depende de: —
Paraleliza com: 10

Descrição:
Bucket S3 para gravações do Egress, com evento de ObjectCreated
configurado para disparar a Lambda do VCA.

Escopo técnico:
- Bucket S3: {env}-voice-call-recordings (ou path dentro de bucket
  existente: s3://{bucket-existente}/voice-recordings/)
  - Decisão: consultar DevOps sobre padrão (bucket dedicado vs path)

- IAM Role do Egress (task 10) com s3:PutObject no bucket/prefix

  para seek no player, adicionar CORS nesse momento.

Critérios de aceite:
- Egress grava .ogg no S3 usando IAM Role (sem credenciais no código)
- Arquivo novo no S3 com prefix recordings/ e suffix .ogg dispara Lambda
- Lambda recebe evento com bucket e key corretos
- Lifecycle policy configurada (IA após 30d, expiração após 365d)
