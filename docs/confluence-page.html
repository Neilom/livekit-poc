Visão Geral

Serviço Node.js/TypeScript que conecta chamadas de voz entre um browser e o WhatsApp (via Meta Cloud API) usando LiveKit como SFU (Selective Forwarding Unit) central.

O bridge funciona como um participante server-side no room LiveKit. Ele recebe o áudio do browser via LiveKit, converte PCM para Opus/RTP e envia para a Meta via WebRTC (werift). No caminho inverso, recebe RTP/Opus da Meta, decodifica para PCM e publica no room LiveKit.

API REST

POST /call/start

Inicia uma chamada para um número WhatsApp.

json Request json Response 200

Fluxo interno:

  1. Cria room LiveKit (call-{uuid}) com emptyTimeout: 300s
  2. Gera token JWT para "browser-agent" com grants de publish/subscribe
  3. Chama createBridge(roomName):
    • Gera token JWT para "bridge-agent"
    • Conecta ao LiveKit como bridge-agent via @livekit/rtc-node
    • Publica LocalAudioTrack ("bridge-audio") com AudioSource (48kHz, 1ch)
    • Cria RTCPeerConnection (werift) com codec Opus (PT 111)
    • Configura transceiver sendrecv com handlers bidirecionais
    • Gera SDP offer e aguarda ICE gathering (timeout 10s)
  4. Envia SDP offer ao Meta via POST /{phoneNumberId}/calls com action "connect"
  5. Armazena sessão em memória e retorna callId + token + sdpOffer + metaCallId

POST /call/:callId/answer

Aplica o SDP answer recebido da Meta.

json Request json Response 200

Fluxo interno:

  1. Normaliza SDP answer (literal \r\n → CRLF real)
  2. Chama pc.setRemoteDescription(answer) no werift
  3. Altera status da sessão para active
  4. Inicia gravação via Egress (se S3 configurado) — startRoomCompositeEgress com audioOnly: true, formato OGG

POST /call/:callId/terminate

Encerra a chamada completamente.

json Response 200

Fluxo interno:

  1. Para gravação via Egress (stopEgress)
  2. Termina chamada no Meta (POST /{phoneNumberId}/calls com action "terminate")
  3. Fecha bridge: pc.close(), localTrack.close(), audioSource.close(), room.disconnect()
  4. Deleta room do LiveKit (deleteRoom)
  5. Remove sessão do Map

GET /call/:callId/status

Retorna o status atual da chamada.

json Response 200

Status possíveis: creating | waiting_answer | connecting | active | terminated | error

GET /config

Retorna configuração pública para o frontend.

json Response 200

Fluxo de Áudio (Bidirecional)

Browser → WhatsApp

  1. Browser → LiveKit: Microfone capturado via livekit-client, publicado como track WebRTC para o SFU.
  2. LiveKit → Bridge: @livekit/rtc-node se inscreve no track do browser-agent e recebe AudioFrames PCM (48kHz, mono, frames de 20ms).
  3. Bridge → Meta: PCM é codificado em Opus via @discordjs/opus e empacotado como RTP (RtpPacket) pelo werift. Payload type 111, SSRC do transceiver sender.
  4. Meta → WhatsApp: Meta roteia o áudio para o usuário final via infraestrutura própria do WhatsApp.

WhatsApp → Browser

  1. WhatsApp → Meta → Bridge: RTP Opus packets recebidos pelo werift via onReceiveRtp.
  2. Bridge decode: Opus é decodificado para PCM via @discordjs/opus. Resulta em Int16Array (48kHz, mono).
  3. Bridge → LiveKit: PCM é publicado via AudioSource.captureFrame(AudioFrame) como "bridge-audio" track.
  4. LiveKit → Browser: O SFU entrega o track remoto ao browser via WebRTC. O livekit-client atribui um elemento <audio>.

Fluxo Manual de Teste (SDP Exchange)

O serviço não implementa webhook da Meta — o SDP answer é aplicado manualmente na página de teste.
  1. Digitar o número WhatsApp destino (com código do país, ex: 5511999999999)
  2. Clicar "Start Call" — gera SDP offer e envia à Meta automaticamente
  3. Aguardar o webhook da Meta com o SDP answer (via ferramentas externas como Graph API Explorer)
  4. Colar o SDP answer no campo e clicar "Apply SDP Answer"
  5. A chamada fica ativa — áudio bidirecional browser ↔ WhatsApp
  6. Clicar "Terminate" para encerrar

Gravação (Opcional)

Se as variáveis S3 estiverem configuradas (S3_ACCESS_KEY, S3_SECRET, S3_BUCKET), a gravação via LiveKit Egress:

  • Inicia automaticamente quando o SDP answer é aplicado (POST /call/:id/answer)
  • Para automaticamente quando a chamada é terminada (POST /call/:id/terminate)
  • Formato: OGG (audio only)
  • Path S3: s3://<bucket>/recordings/<callId>/<roomName>-<timestamp>.ogg

Se S3 não estiver configurado, o serviço funciona normalmente sem gravação — apenas loga um warning.

Detalhes Técnicos do Bridge

Parâmetros de Áudio

ParâmetroValor
Sample Rate48.000 Hz
Canais1 (mono)
Frame Size20ms (960 samples)
CodecOpus (payload type 111)
Formato PCMInt16 (signed 16-bit)

RTP

  • Sequence number: inicializado aleatoriamente, incrementado a cada pacote
  • Timestamp: inicializado aleatoriamente, incrementado por samplesPerChannel (960 por frame de 20ms)
  • SSRC: obtido do transceiver.sender.ssrc do werift

ICE Gathering

  • Timeout: 10 segundos
  • Se timeout, prossegue com candidatos parciais
  • O SDP offer final inclui os ICE candidates coletados

Normalização de SDP

O SDP answer recebido da Meta pode conter escapes literais (\r\n como dois caracteres). A função normalizeSdp() converte para CRLF real e garante terminador \r\n no final.

Limitações e Notas

POC — Não adequado para produção sem modificações
  • Sessões armazenadas em memória — reiniciar o processo perde chamadas ativas
  • @livekit/rtc-node está em Developer Preview
  • Sem webhook da Meta implementado — SDP answer aplicado manualmente
  • Sem autenticação nas rotas REST
  • Sem rate limiting ou circuit breaker
  • O werift é puro TypeScript (sem deps nativas) — funciona em qualquer OS mas pode ter limitações de performance em produção com alto volume
