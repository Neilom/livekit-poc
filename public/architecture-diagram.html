🔊 Fluxo de Áudio (Bidirecional)

01
Browser → LiveKit
Microfone capturado via livekit-client, publicado como track WebRTC para o SFU.
02
LiveKit → Bridge
@livekit/rtc-node se inscreve no track do browser-agent e recebe AudioFrames PCM.
03
Bridge → Meta
PCM é codificado em Opus via @discordjs/opus e enviado como RTP pelo werift ao Meta.
04
Meta → WhatsApp
Meta roteia o áudio para o usuário final via infraestrutura própria do WhatsApp.
05
Caminho Reverso
WA → Meta → werift RTP → Opus decode → PCM AudioSource → LiveKit SFU → Browser.

📋 Ciclo de Vida da Chamada

01
POST /call/start
Cria room no LiveKit, gera tokens JWT (browser-agent + bridge-agent), inicia bridge e envia SDP offer ao Meta.
02
POST /call/:id/answer
Recebe SDP answer do Meta, aplica no werift para estabelecer mídia. Inicia gravação via Egress se S3 configurado.
03
Chamada Ativa
Áudio flui bidirecionalmente. Bridge converte PCM ↔ Opus entre LiveKit e Meta em tempo real (48kHz, mono, 20ms).
04
POST /call/:id/terminate
Para egress, termina chamada no Meta, destrói bridge e deleta room do LiveKit.

🔀 Diagrama de Sequência

Server retorna {status: "terminated"} ao Browser

📁 Mapa de Arquivos

src/server.ts
Express app: CORS, rotas REST, gerencia sessões em Map<callId, CallSession>.
src/call-bridge.ts
Core do bridge: rtc-node + werift + Opus encode/decode bidirecional.
src/livekit-manager.ts
RoomServiceClient: criar/deletar rooms, gerar tokens JWT com grants.
src/meta-api.ts
Fetch para Meta Graph API: connect (SDP offer) e terminate.
src/egress-manager.ts
EgressClient: room composite recording (OGG) para S3.
src/config.ts
Carrega .env, valida vars obrigatórias, exporta config tipado.
src/types.ts
CallStatus, CallSession, CallBridgeHandle e DTOs de resposta.
public/app.js
Client JS: conecta ao LiveKit, publica mic, gerencia fluxo SDP.
